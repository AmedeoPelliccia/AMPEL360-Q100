## Cross-Reference Guide Overview

This training module provides comprehensive guidance on creating and managing 
                    cross-references within the AMPEL360 Q100 documentation ecosystem. Proper 
                    cross-referencing ensures traceability, maintains document integrity, and 
                    facilitates navigation across the complex network of technical documentation.

Purpose: To standardize reference 
                    management practices across SSOT artifacts, S1000D data modules, and DPP integration.

Scope: Internal references, cross-DM 
                    references, SSOT artifact references, DPP integration, traceability matrices, 
                    and CI/CD validation.


## Internal References (internalRef)

Internal references link to content within the same S1000D Data Module using 
                    the internalRef element.

Syntax:

<internalRef internalRefId="target-id" 
             internalRefTargetType="irtt01">
  Link Text
</internalRef>

Where:
  target-id = Unique ID of target element (@id attribute)
  irtt01 = Inline reference type (S1000D standard)

Example:

<!-- Define target -->
<para id="term-bwb">
  BWB (Blended Wing Body) is an aircraft configuration...
</para>

<!-- Reference from elsewhere -->
<para>
  For more information, see 
  <internalRef internalRefId="term-bwb" 
               internalRefTargetType="irtt01">BWB definition</internalRef>.
</para>

Best Practices:

Use descriptive ID values (term-bwb, fig-fuel-cell-diagram)
                        Ensure all target IDs are unique within the DM
                        Use consistent naming conventions for IDs
                        Validate all internal references before publication

## SSOT Artifact References

SSOT artifacts (requirements, analyses, tests) use URI-based references for 
                    cross-linking.


### KNU URI Scheme

knu://{KNU-ID}[#{section}]

Examples:
  knu://KNU-00-00-001-REQ-001
  knu://KNU-28-00-001-REQ-001#section-3.2
  knu://KNU-71-00-001-ICD-001#interface-definition


### Term URI Scheme

term://{term-id}

Examples:
  term://bwb
  term://lh2
  term://fuel-cell
  term://learning-assurance


## DPP Integration References

Digital Product Passport (DPP) integration uses ampel360: URI scheme for 
                    component and data references.

URI Patterns:

ampel360:term:{term-id}
ampel360:unit:{unit-id}
ampel360:component:{serial-number}
ampel360:batch:{batch-id}

Examples:
  ampel360:term:bwb
  ampel360:unit:kilogram
  ampel360:component:FC-001-2026-001
  ampel360:batch:LH2-TANK-BATCH-2026-Q1

Usage in S1000D:

<externalPubRef>
  <externalPubRefIdent>
    <externalPubCode>ampel360:term:bwb</externalPubCode>
  </externalPubRefIdent>
</externalPubRef>


## Traceability Matrix Usage

Traceability matrices establish formal relationships between requirements, 
                    designs, tests, and verification evidence.

Matrix Structure:



| Requirement ID | Design KNU | Test KNU | Verification Evidence | Status |
| --- | --- | --- | --- | --- |
| KNU-28-00-001-REQ-001 | KNU-28-00-001-ICD-001 | KNU-28-00-001-TEST-001 | TR-28-001-v1.2.pdf | VERIFIED |

Mandatory Traces:

REQ → ICD (requirement to design)
                        REQ → TEST (requirement to verification)
                        ICD → PUB (design to publication)
                        TEST → Evidence (test to results)

## Best Practices for Reference Integrity

Follow these practices to maintain reference integrity:

Use Stable IDs: Never 
                            change IDs after publication. If renaming is necessary, create new ID 
                            and maintain redirect.
                        
                        
                            Validate Before Commit: 
                            Run local validation before committing. Use provided validation scripts 
                            in tools/ directory.
                        
                        
                            Document Dependencies: 
                            In KNU YAML frontmatter, list all parent and child KNUs, related KNOTs, 
                            and ATA traces.
                        
                        
                            Update Traceability Matrices: 
                            When creating or modifying REQ/ICD/TEST KNUs, update corresponding 
                            traceability matrix.
                        
                        
                            Use Relative Paths in SSOT: 
                            For SSOT-to-SSOT references, use relative filesystem paths when 
                            possible to support repository reorganization.


## Common Mistakes and Corrections



| Mistake | Problem | Correction |
| --- | --- | --- |
| Hardcoded absolute paths | Breaks on repository reorganization | Use relative paths or URI schemes |
| Missing target IDs | References break silently | Always define @id on referenceable elements |
| Duplicate IDs | Ambiguous references | Ensure all IDs unique within DM scope |
| Broken cross-DM refs after DMC update | References point to old version | Update all referencing DMs when DMC changes |
